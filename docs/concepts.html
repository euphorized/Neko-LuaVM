Architecture

neko runs in its own process. The host process — typically a game — runs untouched. The bridge between them is a kernel-level memory channel that the runtime treats as a memory bus.

Layers

Every interaction with the host goes through three layers, top to bottom:

  1. Lua VM — a stock Luau VM (compiler + interpreter, statically linked) running script code in the runtime's process.
  2. Host bridge — typed read/write helpers that translate Luau objects (Instance, Vector3, etc.) into host-side memory operations.
  3. Memory channel — a kernel-mediated transport that performs RPM/WPM against the host with bounded latency.

Scripts only see the top layer. Everything below is invisible from Luau and unreachable from the host.

State resolution

The host's lua_State isn't a pointer the runtime knows ahead of time — it has to be resolved every session. The resolver walks the host's ScriptContext, decodes a XOR-obfuscated pointer, and validates the result through three independent shape checks before caching.

Once cached, every getgc and gctab call reuses the resolved state. If the host re-creates its state during a server transition, the resolver is invalidated and re-runs lazily on the next call.

Memory model

All host-side reads and writes are proxied through the memory channel. There is no shared address space.

Pointers returned from the runtime (e.g. the address field of a getgc record) are valid in the host's address space, not the runtime's. They're returned as hex strings to make this visible at the API surface — accidentally dereferencing one as a Luau value is a type error, not a crash.

Operation Mode Notes
read<T> RPM Single typed read. Returns zero-initialized T on failure.
rpm RPM Bulk read into a buffer. Used for batched page walks.
wpm WPM Bulk write. Returns false on partial writes.
is_valid Probe Cheap address-range check. No actual read.

Threading

The VM runs on a dedicated worker thread. Scripts execute synchronously on that thread. task.spawn queues a new closure on the same thread; it does not start an OS thread.

Drawing is rendered on the runtime's own swap chain in its overlay window, not in the host. Any draw call from a script is a buffered command that flushes during the next overlay frame — no cross-process drawing.

Lifetimes

Scripts can be re-run any number of times during a session. Between runs, the VM tears down all instance caches, signal connections, drawings, and tween bindings. Globals you defined in a previous run are gone.

What survives:

  • The Luau state itself (held by the runtime, not the script).
  • Drawings created in a previous run if you didn't call :Remove(). They render until the runtime exits.
  • The cached lookup of the host's lua_State.

Failure modes

Three things can go wrong, in order of severity:

Stale pointer. A getgc address is valid at the moment of the call but freed before you write to it. The write either lands harmlessly in another GC object or fails the is_valid check. Detect this by re-running gctab.refresh and checking that the slot still matches the cached interned string.

Resolver miss. The host's lua_State can't be located — usually because the host hasn't finished spinning up its scripting context. The runtime returns nil from getgc_state; scripts should treat this as "host not ready" and back off.

Channel disconnect. The kernel bridge drops. All reads return zero, all writes return false. The runtime continues to accept scripts but they will produce empty results until the channel reattaches.

No assertions about host integrity.

The runtime never assumes that a value at a given offset is what it expects. Every read is followed by a structural validation pass. Mutations refuse to land on slots that no longer hold the expected type.
